## 1. MODULE LIFECYCLE

### 1.1 Before writing any code for a new module
- [ ] Check the ecosystem skill (gmp-pharma-ecosystem) — is this REPLACE or INTEGRATE?
  - REPLACE: build natively as a new module under app/modules/<name>/
  - INTEGRATE: add a connector to core/integration/ — do NOT build a module
- [ ] Write the URS (User Requirement Specification) in docs/urs/<module>.md first
- [ ] Define all database tables and columns before any API code
- [ ] Identify all regulatory requirements for this module (21 CFR Part 11? ALCOA+? GxP?)
- [ ] Add the module name to MODULE_NAMES in test_architecture_boundaries.py

### 1.2 Mandatory files for every new module
```
app/modules/<name>/
  __init__.py
  models.py      — SQLAlchemy models only (no business logic)
  schemas.py     — Pydantic request/response schemas
  router.py      — FastAPI router (no business logic, calls services only)
  services.py    — All business logic (called by router)
  tasks.py       — Overdue/scheduled checker hook
```

### 1.3 After writing the module
- [ ] Register router in api/v1/router.py
- [ ] Import models in main.py (with # noqa) for SQLAlchemy discovery
- [ ] Register tasks hook in main.py lifespan
- [ ] Wire hook to NotificationService
- [ ] Write Alembic migration
- [ ] Write at minimum: schema validation tests + boundary tests + 3 API endpoint tests
- [ ] Run quality gates: pytest test_architecture_boundaries.py + python chaos.py
- [ ] Update TASK_QUEUE.md (mark task done, add any discovered subtasks)

## 2. DATABASE DESIGN

### 2.1 Primary keys
- ALL tables use UUID primary keys (String(36)), generated by the application layer
- Use `uuid.uuid4()` in the service layer — never rely on database UUID generation
- Column name: always `id`
- Pattern:
  ```python
  import uuid
  id: Mapped[str] = mapped_column(String(36), primary_key=True, default=lambda: str(uuid.uuid4()))
  ```

### 2.2 Timestamps — mandatory on every table
Every table must have:
```python
created_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), nullable=False,
                                              server_default=func.now())
updated_at: Mapped[datetime] = mapped_column(DateTime(timezone=True), nullable=False,
                                              server_default=func.now(), onupdate=func.now())
```
- Always timezone=True
- Always UTC
- Audit trail tables: created_at only (no updated_at — they are immutable)

### 2.3 Site isolation — mandatory on every non-foundation table
```python
site_id: Mapped[str] = mapped_column(String(36), ForeignKey("sites.id"), nullable=False, index=True)
```
Every query that returns data to a user must filter by site_id.
No query may return data from a different site than the requesting user's site.

### 2.4 Naming conventions
- Tables: snake_case plural (batch_records, test_results, training_assignments)
- Columns: snake_case (performed_at, target_completion_date)
- Indexes: idx_{table}_{column} (idx_batch_records_site_id)
- FK constraints: fk_{table}_{column} (fk_batch_records_site_id)
- Unique constraints: uq_{table}_{column} (uq_batch_records_batch_number)

### 2.5 Index requirements — MANDATORY
Add an index for:
- Every ForeignKey column (site_id, user_id, etc.)
- Every column used in WHERE clauses in the application
- Every column used in ORDER BY in paginated queries
- Composite index for (site_id, status) where queries filter on both
```python
site_id: Mapped[str] = mapped_column(String(36), ForeignKey("sites.id"), nullable=False, index=True)
```

### 2.6 Soft delete — never hard delete GMP records
GMP records are NEVER hard deleted. Use:
```python
is_deleted: Mapped[bool] = mapped_column(Boolean, default=False, nullable=False)
deleted_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
deleted_by_id: Mapped[str | None] = mapped_column(String(36), ForeignKey("users.id"), nullable=True)
```
All queries must include `WHERE is_deleted = false` by default.

### 2.7 Cross-module references
- Always plain String(36) — never ForeignKey across module boundaries
- Column suffix: `_id` (linked_deviation_id, source_batch_id)
- Document the reference in a comment: `# Loose ref to deviations.id in QMS module`

### 2.8 Partitioned tables (HIGH-VOLUME only)
Tables expected to exceed 10 million rows must be partitioned:
- audit_events: partition by RANGE on created_at (monthly)
- test_results: partition by RANGE on tested_at (monthly)
- batch_record_steps: partition by RANGE on performed_at (monthly)
- env monitoring results: partition by RANGE on recorded_at (monthly)

Declare in Alembic migration with PostgreSQL partitioning syntax.

### 2.9 NULL handling
- Use `nullable=False` by default — be explicit when nullable=True is needed
- Document WHY a column is nullable when it is
- Never store empty strings — use NULL or a meaningful default

### 2.10 Enumerations
- Use string columns with documented allowed values — NOT Python Enum or PostgreSQL ENUM
- Comment the allowed values inline:
  ```python
  status: Mapped[str] = mapped_column(String(50), nullable=False, default="pending")
  # Statuses: pending | in_progress | completed | overdue | waived
  ```
- This avoids Alembic migration pain when adding new status values

## 3. API DESIGN

### 3.1 URL structure
```
/api/v1/{module}/{resource}          — list + create
/api/v1/{module}/{resource}/{id}     — get + update + soft-delete
/api/v1/{module}/{resource}/{id}/{action}  — workflow actions (approve, sign, release)
```
Examples:
- GET  /api/v1/qms/capas
- POST /api/v1/qms/capas
- GET  /api/v1/qms/capas/{id}
- POST /api/v1/qms/capas/{id}/close
- POST /api/v1/mes/batch-records/{id}/release  (requires e-signature)

### 3.2 Pagination — mandatory on ALL list endpoints
No list endpoint may return unbounded results. Always paginate:
```python
@router.get("/capas", response_model=Page[CAPAResponse])
async def list_capas(
    page: int = Query(1, ge=1),
    page_size: int = Query(20, ge=1, le=100),
    site_id: str = Depends(get_current_site),
    ...
```
Maximum page_size: 100. Default: 20.

### 3.3 Response envelope
All list endpoints return:
```json
{
  "items": [...],
  "total": 142,
  "page": 1,
  "page_size": 20,
  "pages": 8
}
```

### 3.4 Error responses
Use standard HTTP codes. Always return a structured error body:
```json
{
  "detail": "Step already recorded. Back-filling is not permitted (ALCOA Contemporaneous).",
  "code": "MES_STEP_BACKFILL_DENIED",
  "field": "performed_at"
}
```
Error code format: {MODULE}_{ENTITY}_{REASON} in UPPER_SNAKE_CASE.

### 3.5 Authentication — every endpoint
Every endpoint must declare a dependency on the current user:
```python
current_user: User = Depends(get_current_active_user)
```
No public endpoints except /health and /api/docs (debug only).

### 3.6 Authorization — check roles, not just authentication
After authenticating, verify the user has permission:
```python
require_permission(current_user, "qms:capa:create")
```
Permission code format: {module}:{resource}:{action}

### 3.7 Site isolation — enforced at router level
Every query must filter by the current user's site_id. The router must pass site_id to every service call. No service may query cross-site data unless the user has a multi-site role.

### 3.8 Audit logging — mandatory on all write operations
Every POST, PATCH, PUT, DELETE must create an AuditEvent record via AuditService:
```python
await AuditService.log(
    session=session,
    user_id=current_user.id,
    action="qms.capa.created",
    entity_type="CAPA",
    entity_id=capa.id,
    changes=diff,
)
```

### 3.9 Input validation
- All request bodies validated via Pydantic schemas
- String length limits on every field (match database column lengths)
- Enum validation for status/type fields
- Date validation: future dates only where required (e.g. target_completion_date)

## 4. SERVICE LAYER

### 4.1 What goes in services.py (not in router.py)
- All database queries
- All business logic and validation
- All cross-module UUID lookups (ReferenceResolver pattern)
- All notification dispatch calls
- All e-signature verification

### 4.2 What stays in router.py
- FastAPI dependencies (auth, site, pagination)
- HTTP request/response transformation
- Calling service functions
- Nothing else

### 4.3 Transactions
- Use `async with session.begin()` for operations that write to multiple tables
- Audit log must be written in the SAME transaction as the business record
- If the business record fails, the audit log must also roll back (not the reverse)

### 4.4 Service function signature pattern
```python
async def create_capa(
    session: AsyncSession,
    data: CAPACreate,
    created_by_id: str,
    site_id: str,
) -> CAPA:
```
- Always accept `session` as first arg
- Always accept `site_id` explicitly
- Never query a database inside a router function

### 4.5 Cross-module reference validation
When storing a cross-module UUID (e.g. linked_deviation_id in MES), validate the referenced record exists before saving:
```python
# Not a FK — validate at application layer
await ReferenceResolver.validate_exists(
    session=session,
    table="deviations",
    record_id=linked_deviation_id,
    site_id=site_id,
)
```

## 5. GMP COMPLIANCE RULES

### 5.1 ALCOA+ implementation map

| Principle | Implementation |
|---|---|
| Attributable | Every record has created_by_id + performed_by_id, linked to users.id |
| Legible | All data stored as structured fields — no free-text blobs for critical data |
| Contemporaneous | Server-set timestamps; back-fill prevention in service layer |
| Original | Append-only records; corrections create new records referencing originals |
| Accurate | Server-enforced acceptance criteria checks; no client-trusted values |
| Complete | NOT NULL on critical fields; mandatory step completion before batch release |
| Consistent | Status flags updated by the system that owns them (scheduler or API) |
| Enduring | Soft delete only; audit_events never deleted; 7-year retention minimum |
| Available | Pagination on all queries; no unbounded data loads |

### 5.2 21 CFR Part 11 requirements per feature

| Feature | Part 11 Requirement |
|---|---|
| Login | Unique user credentials; session timeout; concurrent session limits |
| E-signatures | Re-enter username + password (not session reuse); meaning + date captured |
| Audit trail | Immutable; captures who, what, when, original value, new value |
| Access control | Role-based; system access limited to authorised individuals |
| System validation | IQ/OQ/PQ documented; GAMP 5 Category 5 |
| Record integrity | No back-dating; no silent alteration; checksums on critical records |

### 5.3 Back-fill prevention — mandatory on all execution records
Any field that represents "what happened at a point in time" must reject updates:
```python
if step.performed_at is not None:
    raise HTTPException(400, "Step already recorded. Back-filling is not permitted.")
```
Apply to: BatchRecordStep.performed_at, TestResult.tested_at, CalibrationRecord.performed_at

### 5.4 E-signature required actions
The following actions MUST use ESignatureService.sign() — not just authentication:
- MBR approval
- Batch record release
- CAPA closure
- Deviation closure with root cause
- SOP approval
- Training completion (read & understood)
- OOS investigation final disposition
- Equipment qualification approval

### 5.5 Audit trail — what must be captured
Every audit event must capture:
- user_id, username (denormalized for permanence)
- timestamp (server UTC)
- action code (module.entity.action)
- entity_type, entity_id
- ip_address, session_id
- before_value (JSON), after_value (JSON) for updates

## 7. MIGRATIONS

### 7.1 One migration per logical change
- Do not bundle unrelated changes in one migration
- Migration name format: YYYYMMDD_description_of_change.py
- Example: 20260424_add_notification_hooks_to_overdue_checkers.py

### 7.2 Migration checklist
- [ ] upgrade() is correct and idempotent
- [ ] downgrade() exists and actually reverses the upgrade
- [ ] New columns with NOT NULL have a server_default or are added in 2 steps
- [ ] New indexes are created CONCURRENTLY (PostgreSQL) to avoid table locks
- [ ] Tested on a copy of the dev database before committing

### 7.3 Adding NOT NULL columns to existing tables
Never add a NOT NULL column without a default in one migration — it will fail on a non-empty table.
Two-step pattern:
```python
# Migration 1: Add with default
op.add_column("table", sa.Column("col", sa.String(50), nullable=True))
op.execute("UPDATE table SET col = 'default_value' WHERE col IS NULL")
# Migration 2: Add NOT NULL constraint
op.alter_column("table", "col", nullable=False)
```

### 7.4 Renaming columns
Never rename a column without: (1) adding new column, (2) backfill data, (3) update app, (4) drop old column.
Do this across 4 separate deployments for zero-downtime production changes.

### 7.5 Failure logging — never silent
```python
# WRONG
try:
    op.drop_constraint(...)
except Exception:
    pass

# RIGHT
try:
    op.drop_constraint(...)
except Exception as exc:
    logger.warning("Could not drop constraint: %s", exc)
```

## 8. BACKGROUND TASKS

### 8.1 Hook pattern (mandatory)
```python
# In module tasks.py:
async def check_overdue_{entity}s() -> dict:
    # 1. Query overdue records (SQL COUNT — no full table loads)
    # 2. Update status flags in the database
    # 3. Call NotificationService for each overdue record (or batch notification)
    # 4. Return {"overdue": count, "notified": notified_count}

# In main.py lifespan:
register_overdue_hook("{module}_overdue_{entity}s", check_overdue_{entity}s)
```

### 8.2 SQL-only queries in hooks (no Python-side filtering)
```python
# WRONG — loads entire table
result = await session.execute(select(CalibrationRecord))
rows = result.scalars().all()
overdue = [r for r in rows if r.next_calibration_due < now]

# RIGHT — filter in database
count = await session.scalar(
    select(func.count(CalibrationRecord.id)).where(
        CalibrationRecord.next_calibration_due < now,
        CalibrationRecord.next_calibration_due.is_not(None),
        CalibrationRecord.is_overdue == False,
    )
)
```

### 8.3 Status flag updates — mandatory
After counting overdue records, update their status in the same hook call:
```python
await session.execute(
    update(TrainingAssignment)
    .where(
        TrainingAssignment.due_date < now,
        TrainingAssignment.status == "pending",
    )
    .values(status="overdue", updated_at=now)
)
await session.commit()
```

### 8.4 Notification dispatch — mandatory
```python
if overdue_count > 0:
    await NotificationService.send_rule_based(
        session=session,
        rule_code="{module}_{entity}_overdue",
        context={"count": overdue_count, "site_id": site_id},
    )
```

### 8.5 Distributed scheduler lock (required before multi-worker deployment)
The scheduler must not run in multiple workers simultaneously. Use PostgreSQL advisory lock:
```python
async def _acquire_scheduler_lock(session: AsyncSession) -> bool:
    result = await session.execute(text("SELECT pg_try_advisory_lock(42)"))
    return result.scalar()
```
Only start the scheduler if the lock is acquired.

## 9. INTEGRATION LAYER

### 9.1 All external connections through IntegrationConnector
Never hardcode API calls to DeltaV, SAP, CDS instruments, or any external system in module code.

All external system connections must:
1. Have an `IntegrationConnector` record in the database (configurable, not hardcoded)
2. Use `IntegrationDataFeed` to define what data flows in and out
3. Log every call to `IntegrationEventLog` (success + failure)
4. Be retryable with exponential backoff
5. Fail gracefully — never let an integration failure halt a GMP operation

### 9.2 Circuit breaker pattern
For integrations that call external APIs:
```python
# If 3 consecutive calls fail, mark the connector as degraded
# Alert on degraded status
# Stop attempting calls until manually re-enabled or auto-retry window passes
```

### 9.3 Data transformation
- All incoming data from external systems must be validated and transformed before storage
- Never store raw external payloads directly in business tables
- Use a staging table or transformation layer

### 9.4 Real-time vs. batch
- Hardware sensors (ENV monitoring, CDS instruments): use webhook push or polling, store raw readings, transform asynchronously
- ERP sync (SAP): batch sync every N hours via scheduled job, not real-time
- DCS process data: read-only historian query, never write to DCS

## 10. PERFORMANCE

### 10.1 No N+1 queries — ever
If you're querying related records in a loop, use eager loading:
```python
# WRONG
capas = await session.execute(select(CAPA))
for capa in capas.scalars():
    actions = await session.execute(select(CAPAAction).where(CAPAAction.capa_id == capa.id))

# RIGHT
capas = await session.execute(
    select(CAPA).options(selectinload(CAPA.actions))
)
```

### 10.2 Pagination on every list query
Every query that returns multiple rows must have LIMIT and OFFSET.
Use `page` and `page_size` parameters. Maximum page_size: 100.

### 10.3 Index every FK column
Every ForeignKey column must have `index=True`.
Every column used in a WHERE clause in services.py must have an index.
Every column used in ORDER BY in paginated queries must have an index.

### 10.4 Connection pool — mandatory configuration
```python
# In core/database.py
engine = create_async_engine(
    settings.DATABASE_URL,
    pool_size=20,
    max_overflow=40,
    pool_timeout=30,
    pool_recycle=3600,
    pool_pre_ping=True,
)
```

### 10.5 Query timeout
Set a statement timeout on all queries: 30 seconds maximum.
```python
await session.execute(text("SET statement_timeout = '30s'"))
```

### 10.6 Read-heavy endpoints
For reporting dashboards and compliance reports:
- Use read-only database sessions (if read replica is configured)
- Cache results with a 5-minute TTL (Redis) where regulatory accuracy allows

## 11. OBSERVABILITY

### 11.1 Structured logging — every log must be structured
```python
logger.info(
    "Batch record released",
    extra={
        "module": "mes",
        "action": "batch_record.released",
        "batch_record_id": batch_record.id,
        "released_by_id": user.id,
        "site_id": site_id,
    }
)
```
Never use f-strings for log messages with dynamic data (use `extra` dict).

### 11.2 Required health endpoints
- GET /health — basic liveness (already exists)
- GET /health/scheduler — scheduler status (already exists)
- GET /health/db — database connectivity check
- GET /health/integrations — status of all IntegrationConnectors

### 11.3 Metrics to expose (add when Prometheus is wired)
- Request count per endpoint + status code
- Request latency p50/p95/p99
- Background task execution time per hook
- Queue depth (if message queue is added)
- Active batch records count
- Overdue items count per module

### 11.4 Log levels
- DEBUG: detailed flow tracing (dev only)
- INFO: normal operations, state transitions
- WARNING: recoverable issues (integration degraded, constraint not found in migration)
- ERROR: operation failed, user impact possible (caught exceptions)
- CRITICAL: data integrity at risk (audit log failed, e-signature service down)

## 12. SECURITY

### 12.1 RBAC — permission codes
Every new resource needs permission codes registered:
```
{module}:{resource}:read
{module}:{resource}:create
{module}:{resource}:update
{module}:{resource}:delete
{module}:{resource}:{action}   (e.g. qms:capa:close)
```

### 12.2 Input sanitisation
- Pydantic validates all inputs — never trust raw request data
- String length limits match database column sizes
- No raw SQL queries — use SQLAlchemy ORM or text() with bound parameters only
- File uploads: validate MIME type, scan for malware (virus scanner integration), store outside webroot

### 12.3 Rate limiting
- Login endpoint: max 5 attempts per minute per IP
- All other endpoints: max 100 requests per minute per user
- File upload endpoints: max 10 per minute per user

### 12.4 Session security
- JWT access token: 15-minute expiry
- JWT refresh token: 8-hour expiry (1 shift)
- Refresh tokens are single-use and stored hashed
- Concurrent sessions: configurable per role (default: 3)
- Session terminated on password change

### 12.5 Sensitive data
- Passwords: bcrypt with cost factor 12
- E-signature credentials: re-verified against hashed password — never stored post-verification
- Audit logs: never log plaintext passwords or tokens

## 13. FRONTEND

### 13.1 TypeScript — strict mode always on
tsconfig.json must have `"strict": true`. No `any` types without a comment explaining why.

### 13.2 API client — auto-generated from OpenAPI spec
Use the FastAPI-generated OpenAPI schema to generate TypeScript types:
```
openapi-typescript backend/openapi.json --output frontend/src/api/types.ts
```
Regenerate whenever backend schemas change. Never hand-write API types.

### 13.3 Forms — validation mirrors backend
Frontend validation must match backend Pydantic validation.
Form errors must display field-level messages using the `field` key from error responses.

### 13.4 GMP UI rules
- All timestamp displays: show UTC time + user's local time
- All e-signature fields: separate password entry — never autofill
- Audit trail views: read-only, no editing UI
- Back-fill prevention: disable timestamp input fields once saved
- Status transitions: show current state, allowed transitions only (not all statuses)

### 13.5 No direct database shortcuts
Frontend never calls internal APIs that bypass the normal auth/audit flow.
No admin backdoors in the UI.

## 14. AUTONOMOUS OPERATION

### 14.1 How Cursor picks up work (read TASK_QUEUE.md)
1. Open TASK_QUEUE.md
2. Find the highest-priority task with status: PENDING
3. Change its status to IN_PROGRESS with your start timestamp
4. Work on it following BUILDING_RULES.md and .cursorrules
5. When complete: mark DONE, write a 3-line summary of what was changed
6. Run quality gates before marking DONE
7. Commit with message format: `feat(module): description` or `fix(module): description`

### 14.2 When Cursor MUST stop and ask (escalation conditions)
Stop and set task to BLOCKED if:
- A rule in .cursorrules or BUILDING_RULES.md conflicts with the task
- The task requires a decision between two architecturally significant approaches
- A dependency is missing (e.g. a service doesn't exist yet)
- A regulatory requirement is unclear
- A schema change would require a multi-step migration affecting live data
- The chaos suite fails after the change

### 14.3 Session discipline
Each session must produce:
1. At least one DONE task in TASK_QUEUE.md
2. Quality gates passing (pytest architecture test + chaos.py)
3. A git commit with a descriptive message
4. Updated TASK_QUEUE.md

### 14.4 What Cursor must NOT do autonomously
- Change the module boundary architecture
- Add a new external dependency (pip package) without flagging it
- Remove or alter an existing migration
- Modify audit trail or e-signature logic
- Disable or weaken a test
- Change authentication or RBAC logic
- Refactor the foundation layer (core/) without explicit instruction

### 14.5 3-day audit cycle
Every 3 days, the GitHub Actions workflow (audit-report.yml) runs automatically.
It generates audit/reports/YYYY-MM-DD-audit.json for Matrix Agent review.
Cursor does NOT need to do anything special — the workflow handles it.
After Matrix Agent reviews, a new set of instructions will appear in TASK_QUEUE.md.
